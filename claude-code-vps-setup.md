# Claude Code на своём VPS — полная инструкция

**Основной способ установки — через готовые скрипты из репозитория `vps-claude-setup/`.** Ручные пошаговые команды вынесены в приложение в конце — туда стоит заглядывать только для понимания, что именно делают скрипты, или если что-то пошло не так и нужно доделать/починить руками.

---

## Пререквизиты

### Домен

1. Купите домен (или используйте существующий) у любого регистратора.
2. В панели управления DNS создайте поддомен, например `claude.вашдомен.ru`.
3. Создайте (или отредактируйте) запись **A** для этого поддомена — значение: IP-адрес вашего VPS.
4. Если панель автоматически создала запись **AAAA** (IPv6) — удалите её, если IPv6 на сервере не настроен.
5. Проверьте распространение записи: [whatsmydns.net](https://www.whatsmydns.net) → введите домен, тип A → должен везде показываться IP сервера. Обычно занимает от нескольких минут до часа.

**Дождитесь полного распространения DNS**, прежде чем запускать второй скрипт (Caddy не сможет автоматически выпустить HTTPS-сертификат, если домен ещё не резолвится в IP сервера).

### VPS

1. **Купите сервер** (Ubuntu 24.04) у любого хостинга с серверами в Европе. Понадобятся: IP-адрес, способ входа (SSH-ключ или пароль от хостинга).

2. **Сделайте снэпшот — точка отката до начала настройки.** В панели хостинга (Timeweb Cloud / Webdock и т.п.) → раздел "Снимки"/"Backups"/"Snapshots" → создать снимок вручную, например с названием "чистая Ubuntu до настройки". Это восстанавливается прямо из панели хостинга, даже если сервер вообще перестанет отвечать по SSH — самая надёжная страховка перед экспериментами.

   (Локальный Timeshift на сервере как дополнительный чекпоинт — рискованная идея на маленьких дисках: он хранит копию на том же диске, что и система, и легко может забить всё свободное место, как уже случилось на практике. Если диск небольшой — обойдитесь снэпшотом хостера.)

3. Если у хостинга не было опции добавить SSH-ключ при заказе сервера — см. [Приложение, шаг A.2](#a2-ssh-ключ) про генерацию и добавление ключа вручную.

## Установка через скрипты

### Шаг 1 — клонировать репозиторий на сервер

Зайдите на сервер по SSH (обычно под `root` при первом входе):

```bash
ssh root@IP-сервера
apt update && apt install -y git
git clone https://github.com/ваш-аккаунт/vps-claude-setup.git
cd vps-claude-setup
chmod +x scripts/*.sh
```

### Шаг 2 — базовая настройка сервера

```bash
./scripts/01-server-setup.sh
```

Скрипт спросит имя нового пользователя, перенесёт на него ваш SSH-ключ, **остановится и попросит подтвердить в отдельном окне терминала**, что вход под новым пользователем работает — только после этого отключит пароль и root-логин. Дальше сам поставит `ufw`, fail2ban, базовый firewall (порт 22), Node.js и Claude Code.

Если что-то упадёт с ошибкой — скрипт покажет, на каком именно шаге (`Шаг N/6`) и почему. Повторный запуск безопасен: скрипт идемпотентен и пропускает уже сделанное.

### Шаг 3 — вход в Claude (вручную, единственный шаг, который нельзя автоматизировать)

```bash
ssh новый-пользователь@IP-сервера
claude
```

Выберите **Claude account**. Ссылку авторизации нужно открыть с не-российского IP:
1. Скопируйте ссылку из терминала.
2. Включите VPN на своём устройстве (разово, на минуту).
3. Откройте ссылку, войдите, скопируйте код.
4. Выключите VPN.
5. Вставьте код в терминал сервера.

### Шаг 4 — веб-панель

```bash
cd vps-claude-setup
./scripts/02-webpanel-setup.sh
```

Спросит домен, поставит CloudCLI, Caddy (авто-HTTPS), откроет 80/443 в ufw, дополнительно проверит и при необходимости доделает SSH-хардненинг из шага 2 (если вдруг он не завершился).

### Шаг 5 — открыть панель

Откройте `https://ваш-домен` в браузере (с любого устройства, VPN не нужен) и **сразу** зарегистрируйте владельца панели — первый вошедший становится единственным владельцем.

## Первый запуск и проверка

1. Проверьте базовую работоспособность: отправьте сообщение в чате панели, убедитесь, что Claude отвечает.
2. Проверьте, что вход по паролю и root по SSH действительно отключены:
   ```bash
   sudo grep -i passwordauthentication /etc/ssh/sshd_config
   sudo grep -i permitrootlogin /etc/ssh/sshd_config
   ```
   Оба должны показывать `no`.
3. Проверьте открытые порты:
   ```bash
   sudo ss -tulpn
   sudo ufw status
   ```
   Наружу должны смотреть только 22, 80, 443.

Готово — панель доступна с любого устройства как обычный сайт.

## Если что-то уже сделали вручную до этого

Если вы (как только что) успели выполнить часть шагов руками, прежде чем перейти на скрипты — ничего страшного:
- Оба скрипта идемпотентны — проверяют, что уже сделано (пользователь создан, пакет установлен, служба запущена), и просто пропускают эти пункты.
- Можно смело клонировать репозиторий и запускать `01-server-setup.sh` — он сам разберётся, что уже готово.
- Если вы уже создали нового пользователя вручную и работаете под ним — просто убедитесь, что у него есть `sudo`, и продолжайте со скрипта `02-webpanel-setup.sh` напрямую (он не требует root, наоборот — требует обычного пользователя).
- Если сомневаетесь, что именно уже сделано — приложение ниже описывает те же действия по шагам, можно свериться.

## Бэкап конфигурации в git

Готовые скрипты (`01-server-setup.sh`, `02-webpanel-setup.sh`, шаблоны Caddy/systemd, README) уже собраны и лежат в репозитории `vps-claude-setup/`. Если сервер придётся пересоздать — просто повторить "Установку через скрипты" выше на новом VPS.

### Первая загрузка файлов в репозиторий (через терминал Mac)

```bash
cd vps-claude-setup
git init
git add .
git status              # проверка: что именно git собрался добавить
git diff --cached       # проверка: что именно попадёт в коммит, построчно
git commit -m "initial setup scripts"
git remote add origin https://github.com/ваш-аккаунт/название-репозитория.git
git branch -M main
git push -u origin main
```

При `git push` GitHub запросит авторизацию (даже для публичного репозитория — пуш требует логина, в отличие от клонирования):
- **HTTPS** — логин: ваш username, вместо пароля — **Personal Access Token** (GitHub: Settings → Developer settings → Personal access tokens → Generate new token, права достаточно `repo`).
- **SSH** — если у вас настроен SSH-ключ в GitHub-аккаунте, используйте `git@github.com:ваш-аккаунт/название-репозитория.git` вместо HTTPS-адреса.

### Обновление файлов после первой загрузки

```bash
git add .
git status
git diff --cached
git commit -m "описание изменения"
git push
```

## Приложение: ручная настройка (без скриптов, для справки)

Этот раздел описывает то же самое, что делают скрипты — по шагам, вручную. Полезно, если хотите понять, что именно происходит "под капотом", или если нужно что-то донастроить точечно после того, как скрипты уже отработали.

### A.1 Установите ufw (если его нет)

На некоторых минимальных образах Ubuntu `ufw` не установлен по умолчанию:
```bash
sudo apt update
sudo apt install -y ufw
```

### A.2 SSH-ключ

Если хостинг не предлагал добавить ключ при заказе сервера — сгенерируйте на своём компьютере:
```bash
ssh-keygen -t ed25519 -C "vps-claude"
```
Добавьте на сервер (способ А, если `ssh-copy-id` доступен):
```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub root@ваш-ip
```
Способ Б — вручную:
```bash
ssh root@ваш-ip
mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "содержимое id_ed25519.pub" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```
Проверьте в новом окне терминала, не закрывая текущую сессию:
```bash
ssh -i ~/.ssh/id_ed25519 root@ваш-ip
```

### A.3 Новый пользователь

```bash
ssh root@ваш-ip
adduser myuser
usermod -aG sudo myuser
mkdir -p /home/myuser/.ssh
cp ~/.ssh/authorized_keys /home/myuser/.ssh/
chown -R myuser:myuser /home/myuser/.ssh
chmod 700 /home/myuser/.ssh && chmod 600 /home/myuser/.ssh/authorized_keys
```
Проверьте вход под новым пользователем в отдельном окне терминала:
```bash
ssh myuser@ваш-ip
sudo whoami   # должно вывести: root
```

### A.4 SSH-хардненинг

Проверить текущее состояние:
```bash
sudo grep -i passwordauthentication /etc/ssh/sshd_config
```
Отключить (только после проверки входа под новым пользователем из A.3):
```bash
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

Базовый firewall:
```bash
sudo ufw allow 22/tcp
sudo ufw enable
```
(Подтвердите `y`, когда спросит про возможный разрыв SSH-соединения.)

fail2ban:
```bash
sudo apt install -y fail2ban
sudo systemctl enable --now fail2ban
```

### A.5 Node.js и Claude Code

```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo bash -
sudo apt install -y nodejs
sudo npm install -g @anthropic-ai/claude-code
claude --version
```

### A.6 Веб-панель (CloudCLI + Caddy + firewall) через промпт для Claude Code

После `claude login`, в терминале, где запущен `claude`, вставьте одним сообщением:

```
Настрой на этом сервере веб-доступ к себе. Сначала спроси у меня домен
(вида claude.мой-домен.ru), затем выполни по шагам:

1. Установи глобально npm-пакет @cloudcli-ai/cloudcli — веб-панель
   CloudCLI (Claude Code UI).
2. Создай systemd-службу для автозапуска панели при старте сервера.
   Слушать она должна только 127.0.0.1:3001 — наружу порт напрямую
   не открывать. Служба должна запускаться от имени текущего
   пользователя с его домашней директорией, иначе она не увидит
   мою авторизацию Claude.
3. Установи веб-сервер Caddy и настрой реверс-прокси: указанный домен
   должен проксироваться на 127.0.0.1:3001. HTTPS-сертификат Caddy
   должен получить и настроить автоматически.
4. Добавь в ufw порты 80 и 443 (порт 22 уже должен быть разрешён —
   проверь перед любыми изменениями firewall, чтобы не потерять
   доступ по SSH).
5. Убедись, что служба панели и Caddy запущены и в порядке (systemctl
   status), покажи вывод `sudo ss -tulpn`, и сообщи мне, что нужно
   немедленно открыть домен в браузере и зарегистрировать владельца
   панели. Задачу не считай выполненной, пока я не подтвержу, что
   зарегистрировался и вижу интерфейс чата.
```

## Дополнительно: локальные снэпшоты через Timeshift

Использовать с осторожностью — на маленьких дисках Timeshift может забить всё свободное место (хранит копию на том же диске, что и система). Если диск небольшой (до ~10-15 ГБ) — лучше обойтись только снэпшотом хостера.

```bash
sudo apt install -y timeshift
sudo timeshift --create --comments "чистая Ubuntu до настройки"
```
Посмотреть снимки: `sudo timeshift --list`. Восстановить: `sudo timeshift --restore` (спросит номер снимка, после восстановления системных файлов — `sudo reboot`).

Если Timeshift уже забил диск — удалить:
```bash
sudo umount /run/timeshift/*/backup 2>/dev/null
sudo rm -rf /timeshift
sudo rm -rf /run/timeshift
df -h
```
